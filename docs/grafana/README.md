# Grafana — Rewind operator observability

Version-controlled dashboards, alert rules, and the scrape config for the Rewind
operator-metrics pipeline (Grafana Cloud). Design + the per-metric rationale live
in [`../architecture/observability.md`](../architecture/observability.md); this
directory is the concrete, reviewable artifact it calls for.

## The pipeline (as deployed)

Each process serves Prometheus text on a dedicated **loopback HTTP/1.1** port; a
node-local **Grafana Alloy** scrapes it and `remote_write`s to Grafana Cloud
Prometheus. Service logs go to Loki via a journald sidecar (`{job="rewind"}`).

| Process | `/metrics` port | Prometheus `job` |
|---|---|---|
| `rewind-worker` | `127.0.0.1:9110` (`REWIND_METRICS_PORT`) | `rewind-worker` |
| `rewind-cp` | `127.0.0.1:9111` (`REWIND_CP_METRICS_PORT`) | `rewind-cp` |
| `rewind-front` | `127.0.0.1:9112` (`REWIND_FRONT_METRICS_PORT`) | `rewind-front` |

Loopback **is** the auth (the node-local Alloy is the only scraper) — there is no
bearer on these ports. The scrape blocks are in
[`alloy-scrape.river`](alloy-scrape.river) (the rewind portion of each node's
`~/.config/alloy/config.alloy`; creds + `remote_write`/`loki.write` are
provisioned out-of-band, not in this repo).

## Import the dashboard

Grafana → Dashboards → **New → Import** → upload `rewind-overview.json` → pick
your Prometheus datasource for the `datasource` variable. (The datasource is a
template variable, so the JSON is portable across stacks.)

## Load the alerts

Grafana Cloud uses the Mimir ruler. With
[`mimirtool`](https://grafana.com/docs/mimir/latest/manage/tools/mimirtool/):

```bash
mimirtool rules load docs/grafana/alerts/rewind-rules.yaml \
  --address https://prometheus-prod-66-prod-us-east-3.grafana.net \
  --id <PROM_USER_ID> --key <GRAFANA_CLOUD_API_KEY>
```

(Alternatively, paste them as Grafana-managed alert rules.) Wire a contact point
for `severity: page` (the consensus/availability wedges) vs `severity: ticket`
(degradations). Every rule has a `for:` — a brief election or reconnect is
normal; the alerts fire only on **sustained** conditions.

**Live deployment (2026-06-26):** these are deployed to the `keeneagle1637`
Grafana stack as **Grafana-managed** rules (folder **Rewind**, group `rewind`,
1m eval) — pushed via the Grafana ruler API with a service-account token, since
that's what the available token scoped to (`alert.rules:write`, not the Mimir
`rules:write` a `mimirtool` load needs). The YAML above is the equivalent
portable Mimir-ruler form, kept as the version-controlled source of truth; the
dashboard JSON is the live dashboard. All 10 evaluate `health=ok`, `inactive`
on a healthy cluster.

## What's covered — and what's not (yet)

The **serving RED signal** is in: `http_requests_total{code}` (rate + error ratio,
all three jobs). Plus the consensus wedge signals (`raft_groups_no_leader`,
`raftnet_peers_unreachable`), CP reconciler health, front connection-setup +
leak canaries, and storage durabilize failures.

Planned follow-ups (the dashboard/alerts extend cleanly when they land):

- **`http_request_duration_seconds`** — request-latency histogram (the **D** of
  RED). Needs per-request start-time plumbing through the h2 stream lifecycle.
- **`blob_*`** — dedicated S3/blob op rate/latency/error metrics. Today
  `kvexp_durabilize_failed_total` is the only storage-degradation proxy.
- **JS handler** invocation/error/duration metrics (worker).
- **raft-rs internal logs** → Loki (currently the engine logger is `Discard`).
